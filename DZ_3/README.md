Задача 1: Обеспечение разработки
Для реализации непрерывной интеграции и поставки (CI/CD) в облачной среде предлагается использовать GitLab как единую платформу, объединяющую систему контроля версий и CI/CD. GitLab полностью удовлетворяет всем перечисленным требованиям:

Облачная система – GitLab доступен как SaaS (gitlab.com) или может быть развёрнут в собственном облаке (self-hosted).

Система контроля версий Git – встроенный Git-репозиторий.

Репозитории на каждом сервисе – для каждого микросервиса создаётся отдельный проект (репозиторий).

Запуск сборки по событию – поддерживаются триггеры на push, merge request, tag и другие события.

Запуск сборки по команде с параметрами – через GitLab API можно запускать pipeline с передачей переменных окружения.

Привязка настроек к каждой сборке – переменные окружения и CI-файлы (.gitlab-ci.yml) хранятся в репозитории и могут быть переопределены на уровне pipeline.

Создание шаблонов для различных конфигураций – использование include в .gitlab-ci.yml позволяет создавать переиспользуемые шаблоны для разных типов сборок.

Безопасное хранение секретов – встроенное хранилище переменных (CI/CD variables) с возможностью маскировки и ограничения доступа, интеграция с HashiCorp Vault.

Несколько конфигураций для одного репозитория – можно определить несколько job’ов с разными правилами запуска (например, сборка приложения, сборка Docker-образа, прогон тестов).

Кастомные ступени сборки – гибкая настройка пайплайнов с произвольными стадиями (build, test, deploy, etc.).

Собственные Docker-образы для сборки – можно использовать любые образы из Docker Hub или собственного registry, задавая image для каждого job’а.

Развёртывание агентов на обычных серверах – GitLab Runner можно установить на любые Linux/Windows-серверы, зарегистрировать в проекте и использовать для выполнения заданий.

Параллельный запуск сборок – настройка параллельного выполнения job’ов с помощью parallel или использование нескольких раннеров.

Параллельный запуск тестов – разделение тестов на группы (например, по классам) с помощью parallel или использование матрицы тестирования.

Обоснование выбора: GitLab предоставляет всё необходимое в одном продукте, что снижает сложность интеграции, упрощает администрирование и обеспечивает единый интерфейс для разработчиков. Гибкость настройки пайплайнов и поддержка широкого спектра языков и инструментов делает его идеальным для микросервисной архитектуры.

Задача 2: Логи
Для централизованного сбора и анализа журналов предлагается стек ELK (Elasticsearch, Logstash, Kibana) с использованием Filebeat в качестве легковесного агента. Это решение закрывает все требования:

Сбор журналов в центральном хранилище – все хосты отправляют логи в Elasticsearch.

Минимальные требования к приложениям – приложения пишут логи в stdout/stderr, Filebeat собирает их из стандартного вывода контейнеров или системных журналов.

Гарантированная доставка – Filebeat поддерживает буферизацию и повторные попытки при сбоях, что обеспечивает доставку «хотя бы один раз».

Поиск и фильтрация – Kibana предоставляет мощный интерфейс с возможностью поиска по произвольным полям, фильтрации, сохранённым запросам.

Пользовательский интерфейс с ограничениями – в Kibana можно настроить роли и разрешения (например, разработчики видят только свои сервисы).

Ссылка на сохранённый поиск – Kibana позволяет сохранять запросы и генерировать прямые ссылки для их быстрого открытия.

Обоснование: ELK является индустриальным стандартом для централизованного логирования. Filebeat потребляет мало ресурсов и легко настраивается. Стек масштабируется горизонтально и поддерживает высокую доступность.

Задача 3: Мониторинг
Для сбора метрик и визуализации состояния хостов и сервисов предлагается связка Prometheus + Grafana. Это решение удовлетворяет всем требованиям:

Сбор метрик со всех хостов – Prometheus использует pull-модель, опрашивая экспортёры на каждом хосте.

Метрики ресурсов хостов – Node Exporter собирает CPU, RAM, HDD, Network.

Метрики ресурсов для каждого сервиса – для контейнеров используется cAdvisor (сбор метрик CPU, RAM, Network, диска) или kube-state-metrics (в Kubernetes).

Специфические метрики сервисов – приложения могут предоставлять эндпоинт /metrics в формате Prometheus, используя клиентские библиотеки.

Пользовательский интерфейс с запросами и агрегацией – Grafana позволяет строить панели с запросами PromQL, агрегировать данные по времени, сервисам, хостам.

Настройка панелей для состояния системы – Grafana предоставляет готовые дашборды (например, для Node Exporter, для микросервисов) и возможность создавать кастомные панели.

Обоснование: Prometheus является стандартом для мониторинга в микросервисной архитектуре благодаря своей гибкости, надёжности и широкой экосистеме экспортёров. Grafana предоставляет богатые возможности визуализации и настройки дашбордов.
